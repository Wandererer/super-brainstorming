# Scoring Rubric (SSOT)

수렴 평가의 단일 진실 원천. 심사 에이전트 프롬프트에는 이 앵커들이 그대로 들어간다.
집계(중앙값·가중치·랭킹·wildcard seat)는 전부 `scripts/validate_ideas.py`가 수행한다 — LLM은 축별 점수만 낸다.

> **research_first 모드 심사 노트**: 카드에 붙은 `evidence_ids`의 근거 강도(strength)와 신선도(freshness)를 impact/timing 판단에 반영한다. 근거가 주장을 실제로 지지하는지 확인하고, **장식용 인용**(근거와 무관한 아이디어)이면 해당 축을 낮게 — red-team이 놓쳤다면 note에 명시한다. 근거가 있다고 novelty를 깎지 않는다: 근거는 출처의 증명이지 모방의 증거가 아니다.

## The 5 Axes (1-5, integers)

### Novelty — 참신성
"이미 있는 것과 얼마나 다른가?"

| Score | Anchor |
|-------|--------|
| 1 | 검색하면 동일한 제품/방법이 여럿 나온다 |
| 2 | 기존 해법의 명백한 변형 (스킨 교체 수준) |
| 3 | 기존 요소의 조합이지만 조합 자체가 새롭다 |
| 4 | 이 접근을 이 문제에 쓴 사례를 찾기 어렵다 |
| 5 | 문제의 프레임 자체를 바꾼다 — 카테고리가 없다 |

### Impact — 임팩트
"성공하면 얼마나 큰가?" (frame의 success_criteria 기준)

| Score | Anchor |
|-------|--------|
| 1 | 성공해도 아무도 눈치 못 챈다 |
| 2 | 소수 사용자의 마이너한 개선 |
| 3 | 타깃 사용자의 핵심 문제 하나를 명확히 해결 |
| 4 | 타깃의 워크플로우/지출을 구조적으로 바꾼다 |
| 5 | 성공 시 카테고리를 정의한다 — "X 이전/이후"가 생긴다 |

### Feasibility — 실현가능성
"**frame의 제약**(자원·기간·자산) 안에서 실행 가능한가?" — 펀딩받은 스타트업 기준이 아니다.

| Score | Anchor |
|-------|--------|
| 1 | 연구 프로젝트 — 핵심 기술/자원이 존재하지 않음 |
| 2 | frame의 제약을 크게 초과 (팀/자금/기간 배수 필요) |
| 3 | 제약 안에서 가능하지만 빠듯하다 — 하나라도 어긋나면 미룬다 |
| 4 | 제약 안에서 무리 없이 MVP 도달 |
| 5 | 보유 자산만으로 몇 주 내 첫 버전 가능 |

### Fit — 적합성
"frame의 자산·목적·anti-goal과 맞물리는가?"

| Score | Anchor |
|-------|--------|
| 1 | anti-goal에 저촉되거나 목적과 무관 |
| 2 | 목적엔 맞지만 보유 자산이 전혀 레버리지 안 됨 |
| 3 | 목적에 맞고 자산 일부를 활용 |
| 4 | 자산이 명확한 불공정 우위(unfair advantage)로 작동 |
| 5 | 이 사람/팀이 아니면 하기 어려운 아이디어 |

### Timing — 시의성
"왜 지금인가?" (**오늘 날짜** 기준 — 프롬프트의 {TODAY})

| Score | Anchor |
|-------|--------|
| 1 | 10년 전에도 가능했고 아무도 안 한 이유가 있다 |
| 2 | 시점과 무관 — 지금이어야 할 이유가 없다 |
| 3 | 순풍이 있다 (관련 트렌드 존재) |
| 4 | 최근 1-2년의 변화(기술 단가·행동·규제)가 문을 열었다 |
| 5 | 지금 아니면 닫히는 창 — 6-12개월 내 선점 우위 |

## Default Weights

| Axis | Weight |
|------|--------|
| impact | 0.30 |
| feasibility | 0.25 |
| novelty | 0.20 |
| fit | 0.15 |
| timing | 0.10 |

- `frame.json`의 `convergence.scoring_weights`로 세션별 오버라이드 가능 (코드가 합=1로 정규화)
- 가중치 조정 가이드: 1인 개발/타임박스 → feasibility ↑, 투자 유치용 → impact·novelty ↑, 레드오션 진입 → novelty·timing ↑

## Judge Protocol

1. **독립성** — 심사위원 2-3인은 서로의 점수를 보지 못한다. 순차 실행하더라도 이전 심사 결과를 프롬프트에 넣지 않는다.
2. **red-team 이후** — 심사위원은 red-team의 objection이 병합된 카드를 받는다 (반론을 알고 채점).
3. **전 축 채점** — 모든 후보의 5축 전부. 누락은 게이트가 exit 2로 잡는다.
4. **앵커 준수** — 취향이 아니라 위 앵커 문장 기준. `note`에 결정적 요인 한 줄.
5. **집계는 코드** — 축별 median → 가중합 → 랭킹 → tiebreak(impact→novelty→idea_id) 전부 `validate_ideas.py`.

## Wildcard Seat (코드 강제)

쇼트리스트 상위 N에 `wildness ≥ 4`가 하나도 없으면, 마지막 1석을 wild 후보 중 최고 순위로 교체한다(`wildcard_seat: true` 플래그). "과격한 아이디어 환영"을 수사가 아니라 자리로 보장하는 장치. wild 후보 자체가 없으면 교체 없이 경고만 남는다.

## Kill vs Park (red-team 기준)

| 판정 | 기준 | 이후 |
|------|------|------|
| `killed` | 구조적 불가능: anti-goal 저촉, 하드 제약과 우회 불가 충돌, 획득 불가 자산 의존, 자기모순 메커니즘 | `kill_reason` 필수 → parking lot의 Killed 섹션 (부활 조건 명시) |
| `parked` | 지금 frame에선 아니지만 조건이 바뀌면 유효 (다른 goal_type, 다른 자원) | parking lot의 Parked 섹션 (`revisit_when` 명시) |
| 후보 유지 | "어렵다", "리스크 있다"는 죽을 사유가 아니다 — 심사로 보낸다 | 심사 → 랭킹 |

## Rubber-Stamp Guard

한 심사위원이 5개 이상의 아이디어에 동일한 5축 점수 튜플을 반복하면 게이트가 경고를 남긴다. 경고가 나오면 해당 심사를 폐기하고 새 심사위원을 1회 재실행하는 것을 권장한다.
